1. Product Overview

MFP (UI brand: UniSocial) is a multi-tenant social media management SaaS.

  • User auth with OTP + JWT session lifecycle
  • Organization onboarding and role-based access control
  • Social connections: Meta/Instagram, LinkedIn, YouTube
  • Cross-platform post creation, scheduling, async publishing
  • Analytics aggregation and dashboard reporting
  • Industry news ingestion + feed display
  • AI-assisted post generation
  • Audit log and platform admin operations

4. Security and Session Design

Authentication Pattern

  • Access token: JWT bearer (short TTL: 15 minutes)
  • Refresh token: HttpOnly cookie (default 7 days, remember-me extends to 30 days)
  • Refresh rotation and blacklist enabled via SimpleJWT settings

Why this was implemented

  • Reduces XSS token exposure by keeping refresh token out of JS-accessible storage
  • Balances user convenience and security with short-lived access tokens
  • Supports seamless background refresh and cross-tab consistency

OTP Security

  • OTP stored as HMAC digest, never plain text
  • Attempt limit + cooldown + expiry
  • Unique active OTP constraint per user and purpose

Cookie/CORS/CSRF

  • Environment-driven cookie domain, secure, and SameSite behavior
  • Credentialed CORS enabled
  • Trusted origins configured for CSRF

Credential Protection

  • Provider tokens are encrypted in DB (django-encrypted-model-fields)
  • Password hashes managed by Django auth system

6. How Scheduling Works (Deep Explanation)

How account selection is determined

The scheduler does not guess account/provider. It uses explicit FK chain:

PostPlatform -> PublishingTarget -> SocialAccount -> credentials

Create flow

  1. Frontend sends publishing_target_ids[] + media + caption + schedule time.
  2. Backend validates targets belong to requesting org and are active.
  3. One PostPlatform row is created per target.
  4. Media rows are attached and provider constraints validated.

Dispatch and publish flow

  1. Beat task runs every minute and picks due jobs.
  2. Worker locks row for idempotency (select_for_update).
  3. Provider-specific publisher executes (Instagram/LinkedIn/YouTube).
  4. Success/failure/retry metadata is persisted on each platform row.

7. Social OAuth Flows

Meta/Instagram

  • State signed with HMAC to prevent tampering
  • Short-lived token exchanged for long-lived token
  • Page sync creates Meta and Instagram publishing targets

LinkedIn

  • State stored in cache with TTL
  • Profile fetched from userinfo or decoded from id_token fallback
  • On token revoke/invalid: account and targets deactivated to force reconnect

YouTube

  • OAuth includes offline refresh token access
  • Account/channel persisted and publishing target created
  • Token refresh can happen both scheduled and during publish

8. Analytics Pipeline

  1. Periodic task scans successfully published post-platform rows.
  2. Fetcher chosen from provider registry.
  3. Latest metrics upserted in current table.
  4. Snapshot written with de-duplication window to reduce noise.
  5. Read APIs aggregate and cache responses for dashboard/graphs.

10. Why “Unauthorized / Session Checking / Slow” Happens

Unauthorized & refresh loops

  • Refresh cookie domain/samesite mismatch
  • Stale frontend JS bundle after deployment
  • Expired/blacklisted refresh token
  • Inactive user account now blocked by refresh/login checks

Long “checking session” and endpoint slowness

  • Cold cache, slow DB query, or provider API latency
  • Worker queue backlog
  • Multiple heavy calls triggered on initial page mount
  • Infrastructure resource limits

12. Operational Guidance

  • Keep secrets out of Git and rotate if accidentally exposed.
  • Hard refresh frontend after deployment to avoid stale auth logic.
  • Use health endpoints /health/ and /ready/ for checks.
  • Use Flower + request timing logs to diagnose queue and latency issues.
