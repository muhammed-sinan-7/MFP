1) Executive Summary

MFP (brand: UniSocial) is a multi-tenant social media management SaaS that allows organizations to connect social accounts, schedule posts across platforms, and view analytics from a single workspace.

  • Authentication + OTP + onboarding
  • Organization and role-based access
  • Meta/Instagram, LinkedIn, YouTube account integrations
  • Post scheduling and asynchronous publishing workers
  • Analytics dashboard + per-platform analytics
  • Industry news feed ingestion
  • AI-assisted content generation
  • Audit trail and platform admin control center

2) Product Goals and Why Architecture Is Designed This Way

Business goals

  • Centralize multi-platform publishing in one workspace.
  • Reduce manual effort and platform switching.
  • Give organization-level governance and analytics.
  • Enable safe collaboration via role permissions.

Technical goals

  • Keep interactive APIs fast while provider calls can be slow.
  • Use async processing for publishing and sync workflows.
  • Ensure strict organization data isolation.
  • Support cloud deployment and scaling with containers.

Why Celery + Redis + cache are used

  • Provider APIs (Meta/LinkedIn/YouTube) are network-bound and can timeout.
  • Publishing and analytics sync must continue even if user closes browser.
  • Dashboard endpoints need cache to avoid repeated heavy aggregation.

End-to-end request flow

  1. User opens frontend app domain.
  2. Frontend calls API domain via Axios.
  3. Nginx proxies to Django web service.
  4. Django validates JWT, resolves organization context, executes endpoint logic.
  5. Heavy work is queued to Celery through Redis.
  6. Worker writes status/metrics back to PostgreSQL.

8) Authentication, Session, and Security

JWT/session model

  • Access token in auth header (frontend storage).
  • Refresh token in HttpOnly cookie (backend-managed).
  • Refresh rotation + blacklist enabled.
  • Remember-me extends refresh lifetime.

OTP security model

  • OTP generated random 6-digit code.
  • Stored as HMAC digest (secret-bound, non-reversible).
  • Cooldown and throttles prevent spam.
  • Attempt caps with lock behavior.

Frontend auth resilience (implemented)

  • Public-route bootstrap guard avoids stale protected calls on login page.
  • refreshInFlightPromise dedupes parallel refresh requests.
  • Only refresh 401 forces logout; transient errors do not instantly wipe auth.
  • Cross-tab logout sync via storage events.

CORS/CSRF/cookie controls

  • Credentials enabled for cross-origin cookie refresh flow.
  • Environment-driven cookie domain, secure flag, SameSite mode.
  • Trusted origins and allowed origins configurable via env.

10) Social Account Integration Flows

Meta/Instagram

  • HMAC-signed OAuth state with timestamp validation.
  • Code exchange -> long-lived token storage.
  • Page sync creates Meta targets and Instagram targets.
  • Immediate sync + async sync task for consistency.

LinkedIn

  • Cache-backed state validation.
  • Profile retrieval with timeout fallback to id_token decode.
  • Publish-side token failures mark account inactive and deactivate targets.

YouTube

  • Offline OAuth scopes include upload and analytics read.
  • Channel fetched on callback and target created.
  • Refresh token used periodically and on-demand.

11) Scheduling and Publishing Internals

How scheduler maps to the correct account

Scheduling request uses publishing_target_ids[]. Each ID belongs to one social account, so provider/account is deterministic:

PostPlatform -> PublishingTarget -> SocialAccount -> credentials

Validation controls

  • Target must belong to request org and be active.
  • Scheduled time must be in future.
  • Provider rules (e.g., Instagram media required, YouTube requires video).

Dispatch engine

  • Beat runs every minute.
  • Picks pending jobs and stale processing jobs.
  • Worker applies idempotency lock and provider publisher adapter.
  • Retry/backoff up to 3, then fail with reason.

Provider publishers

  • InstagramPublisher: Graph media creation + publish sequence
  • LinkedInPublisher: asset upload + UGC post publish
  • YouTubePublisher: video insert API publish

12) Analytics Data Pipeline

Sync flow

  1. Find successful post-platform rows.
  2. Select provider fetcher from registry.
  3. Fetch metrics from provider APIs.
  4. Update current analytics table.
  5. Create snapshot if changed or interval allows.

Why current + snapshots both exist

  • Current table serves latest values quickly.
  • Snapshots power charts/trends and fallbacks.

Endpoint behavior

  • Overview/distribution/charts/recent-posts cached per org.
  • Full dashboard endpoint has fallback logic to avoid empty UI when current table sparse.

13) News Ingestion Pipeline

  • Beat task dispatches ingestion every 30 minutes.
  • Runs per industry source set.
  • Persists articles and returns org-industry filtered feed.

14) AI Generation Pipeline

20) Caching and Performance Strategy

  • Redis cache for analytics overview/distribution/charts/dashboard and news endpoints.
  • Frontend short-lived caches for selected service calls (for smoother UX).
  • DB indexes on high-frequency filter fields (provider/status/scheduled timestamps).
  • Async processing to keep API request path responsive.

21) Observability and Health Checks

  • /health/ basic app liveliness

23) Common Production Issues and Root Causes

Unauthorized and forced logout loops

  • Refresh cookie not included due domain/path/samesite/secure mismatch
  • Old frontend JS bundle after deploy
  • Refresh token expired/blacklisted
  • User inactive (login/refresh now blocked by backend checks)

Long session-checking and slow pages

  • Backend latency (DB/network/provider calls)
  • Worker queue saturation
  • Multiple heavy endpoints called simultaneously
  • Cold cache on first requests

24) Developer Runbooks

Runbook A - Connect and schedule

  1. Connect account in /accounts.
  2. Verify targets available via /social/publishing-targets/.
  3. Create schedule from /schedule.
  4. Confirm status transitions pending -> processing -> success/failed.
  5. Check worker logs when failed.

Runbook B - Fix auth refresh issue

  1. Check refresh cookie presence on API domain.
  2. Verify backend cookie settings and CORS origins.
  3. Hard refresh frontend and verify latest bundle hash.
  4. Test /auth/token/refresh/ and inspect status/payload.

Runbook C - Analyze slow endpoint

  1. Inspect slow request logs from middleware.
  2. Check container CPU/memory and worker queue depth.
  3. Check DB query plan and indexes for endpoint filters.
  4. Validate cache hit/miss behavior for repeated requests.
